# Windows 截屏翻译应用开发方案

## 1. 项目目标

开发一个运行在 Windows 桌面的截屏翻译应用，支持以下核心能力：

1. 用户通过快捷键触发截图。
2. 对截图中的文字进行识别与理解。
3. 调用本地视觉大模型完成翻译。
4. 将翻译结果以悬浮窗、贴片或结果面板形式展示。
5. 全流程本地运行，优先保护隐私，减少网络依赖。

结合你提供的批处理脚本，当前模型服务形态已经比较明确：

- 推理引擎：`llama.cpp` 的 `llama-server.exe`
- 模型：`Qwen2.5-VL-7B-Instruct-q4_k_m.gguf`
- 多模态投影：`mmproj-F16.gguf`
- API 地址：`http://127.0.0.1:8081`

这意味着客户端应用不需要直接集成模型推理，只需要负责：

- 启动或检测本地模型服务
- 抓取屏幕图像
- 将图像编码后发送给本地 API
- 解析模型返回结果
- 用适合桌面场景的方式展示翻译结果

## 2. 产品形态建议

推荐优先做一个轻量桌面应用，而不是浏览器扩展或纯命令行工具。

首版建议采用以下交互：

1. 应用后台驻留托盘。
2. 用户按下全局快捷键，例如 `Ctrl + Shift + Q`。
3. 屏幕进入区域截图模式。
4. 截图完成后立即显示“识别中/翻译中”状态。
5. 结果展示在截图区域旁边，支持：
   - 原文
   - 译文
   - 复制文本
   - 重新翻译
   - 切换目标语言

后续增强版再考虑：

- 自动文本框覆盖翻译
- 历史记录
- OCR 专用模式
- 术语表
- 双语对照
- 游戏/漫画/视频字幕场景优化

## 3. 技术路线建议

## 3.1 客户端技术选型

优先推荐：`Tauri + React + TypeScript`

原因：

1. Windows 桌面能力足够，体积比 Electron 更轻。
2. 前端 UI 开发效率高，后续做悬浮窗、设置页、历史页都方便。
3. Rust 后端适合处理：
   - 全局快捷键
   - 截图
   - 系统托盘
   - 窗口置顶
   - 本地文件存储

备选方案：

- `Electron + React`
  - 优点：生态成熟、桌面截图和窗口管理资料多。
  - 缺点：安装包和内存占用更大。

- `PySide6 / Qt for Python`
  - 优点：原生桌面感较强，截图和悬浮窗实现直接。
  - 缺点：后续 UI 迭代和现代化界面开发效率略低。

如果你的目标是“尽快做出可用版”，我建议：

- MVP：`Tauri + React + TypeScript`
- 若你更熟悉 Python：`PySide6` 也完全可行

本方案以下内容默认基于 `Tauri`。

## 3.2 本地模型接入方式

你已经提供了本地模型启动脚本，因此客户端只需要做 HTTP 调用。

建议封装一个 `ModelService`，负责：

1. 检测 `8081` 端口是否可用。
2. 如果不可用，提示用户启动模型，或者由应用代为启动批处理。
3. 将截图转换为 Base64。
4. 按 `llama-server` 的多模态聊天接口格式发送请求。
5. 对模型输出做结构化解析。

建议优先尝试兼容 OpenAI 风格的 `chat/completions` 接口，如果当前 `llama-server` 参数下暴露的是该风格接口，可直接用如下请求思路：

```json
{
  "model": "Qwen2.5-VL-7B-Instruct",
  "messages": [
    {
      "role": "user",
      "content": [
        {
          "type": "text",
          "text": "请识别图片中的文字并翻译成简体中文，只输出译文。"
        },
        {
          "type": "image_url",
          "image_url": {
            "url": "data:image/png;base64,..."
          }
        }
      ]
    }
  ],
  "temperature": 0.2,
  "max_tokens": 512
}
```

如果你本地 `llama-server` 版本接口格式略有差异，需要在开发初期先做一次实际联调确认。

## 5. 关键功能设计

## 5.1 截图流程

推荐流程：

1. 用户按快捷键。
2. 创建全屏透明遮罩窗口。
3. 用户拖拽选择区域。
4. 获取选区位图。
5. 关闭遮罩，弹出等待态。
6. 将图片送入翻译管线。

关键点：

- Windows 高 DPI 下坐标与截图尺寸可能不一致，需要统一逻辑像素和物理像素。
- 多显示器环境下，左侧副屏可能产生负坐标。
- 遮罩窗口需要置顶但不干扰最终结果浮窗。

## 5.2 模型提示词策略

建议不要只写“翻译这张图”，而是做成稳定模板。

建议首版准备 3 种模式：

1. 普通翻译
   - 适合截图中的段落、界面文本、网页内容
2. OCR + 翻译
   - 同时输出原文和译文
3. 严格译文
   - 只输出译文，便于浮窗直接展示

推荐默认提示词：

```text
请阅读图片中的文字内容，并翻译为简体中文。
要求：
1. 保留原意，不要编造图片中不存在的内容。
2. 如果图片中存在多段文本，请按阅读顺序输出。
3. 如果有界面按钮、菜单、标题，请简洁翻译。
4. 默认只输出译文，不要添加解释。
```

增强模式提示词：

```text
请先准确识别图片中的文字，再将其翻译为简体中文。
输出格式：
原文：
<识别出的原文>

译文：
<对应译文>
```

## 5.3 结果展示

MVP 不建议一开始就做“原位置文字替换渲染”，因为这涉及：

- 文本框检测
- 自动排版
- 背景修补
- 多语言换行

首版建议采用：

- 轻量结果卡片
- 支持拖动
- 一键复制
- 固定在截图区域旁边

第二阶段再做：

- 框选文本区域并在原图附近逐块展示
- 图像内覆盖翻译

## 5.4 历史记录

建议本地保存最近 100~500 条记录：

- 时间
- 缩略图
- 原文
- 译文
- 源语言/目标语言
- 耗时

存储方式建议：

- 配置：JSON 或 SQLite
- 历史：SQLite 更合适

如果想先快做 MVP，可先用 JSON 文件。

## 6. API 集成方案

## 6.1 启动检测

应用启动时：

1. 请求 `http://127.0.0.1:8081/health` 或类似探活地址。
2. 如果失败，再尝试请求根路径或模型列表接口。
3. 如果仍失败，提示：
   - 手动启动模型
   - 或点击按钮自动启动批处理

## 6.2 自动启动模型

建议在设置中允许用户填写批处理路径，例如：

- `D:\AI_Models\start_qwen_vl_server.bat`

客户端启动时可以：

- 检测未启动
- 询问是否启动本地模型
- 使用系统进程拉起批处理

注意：

- 首次启动模型可能较慢，需要轮询 10~60 秒。
- 要避免重复启动多个 `llama-server.exe` 实例。

## 6.3 请求封装

客户端应统一封装：

- `checkModelServer()`
- `translateImage(imagePath | imageBytes, options)`
- `cancelCurrentTask()`

返回值建议统一成：

```ts
type TranslationResult = {
  sourceText?: string;
  translatedText: string;
  elapsedMs: number;
  model: string;
  rawResponse?: unknown;
};
```

## 8. 分阶段开发计划

## 阶段 1：技术验证 PoC

目标：

- 能手动选图
- 能调用本地视觉模型
- 能返回翻译结果

交付内容：

1. 截图或图片选择
2. 向本地模型发送图片
3. 输出翻译文本
4. 记录接口格式与实际响应

验收标准：

- 对一张含英文文本的图片，3 到 15 秒内返回中文翻译

## 阶段 2：桌面 MVP

目标：

- 真正形成“快捷键截图 -> 翻译 -> 展示”的闭环

交付内容：

1. 系统托盘
2. 全局快捷键
3. 区域截图
4. 结果悬浮窗
5. 基础设置页

验收标准：

- 在 Windows 10/11 上稳定运行
- 连续使用 20 次无明显崩溃

## 阶段 3：可用性增强

交付内容：

1. 历史记录
2. 多语言切换
3. 提示词模板
4. 自动启动模型服务
5. 错误提示优化

## 阶段 4：高级体验

交付内容：

1. 文本块级翻译展示
2. 原位覆盖翻译
3. 游戏/漫画模式
4. 批量截图翻译

## 9. 风险与难点

## 9.1 截图与 DPI 适配

这是 Windows 桌面工具常见难点，建议尽早验证。

## 9.2 模型输出不稳定

视觉模型可能出现：

- 漏识别
- 多输出解释性文本
- 表格顺序错乱

应对方式：

- 优化提示词
- 增加“严格译文模式”
- 保留原始响应用于调试

## 9.3 模型响应速度

Qwen2.5-VL-7B 在本地推理时，截图面积越大越慢。

建议：

- 优先区域截图
- 对超大图片先压缩
- 将默认输出长度限制在合理范围

## 9.4 批处理启动稳定性

自动拉起 `.bat` 时需要考虑：

- 路径包含空格
- 模型未完全启动就发请求
- 重复进程

## 10. 首版实现建议

如果我们马上开始落代码，建议先做下面这个最小闭环：

1. 初始化 `Tauri + React + TypeScript` 项目。
2. 做一个“选择图片并翻译”的测试页面。
3. 接通 `http://127.0.0.1:8081` 本地模型。
4. 确认图片请求格式和响应格式。
5. 再接入 Windows 区域截图。
6. 最后补系统托盘和全局快捷键。

这样做的好处是：

- 先打通最核心的模型能力
- 尽早发现接口兼容问题
- 避免一开始就陷入桌面交互细节

## 11. 建议的近期任务清单

下一步建议按这个顺序推进：

1. 建立项目骨架。
2. 编写本地模型联调脚本。
3. 记录 `llama-server` 实际接口格式。
4. 完成图片翻译 PoC。
5. 完成截图能力。
6. 完成悬浮翻译结果窗。
7. 增加设置页和历史记录。

## 12. 我对当前需求的默认假设

为了便于先推进方案，我先做了这些默认假设：

1. 目标系统是 `Windows 10/11`。
2. 目标语言默认是“翻译成简体中文”。
3. 模型服务由 `llama-server` 提供 HTTP API。
4. 应用以中文 UI 为主。
5. 你希望优先做可用 MVP，而不是先做复杂的原位图像重绘。

如果这些假设成立，当前方案可以直接进入 PoC 开发阶段。
